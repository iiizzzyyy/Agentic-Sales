Three Architectural Approaches

For hierarchical multi-agent orchestration — choose the foundation pattern

A

Centralized Orchestrator (Hub-and-Spoke)

Single Manager agent owns all client communication. Receives request → analyzes intent → routes to Specialist → aggregates results → responds to client.

Pros

  • Single point of control — easy to reason about
  • Consistent client experience
  • Manager can do multi-turn clarification before delegating
  • Easy to add new specialists

Cons

  • Manager is a bottleneck
  • Single point of failure
  • All traffic flows through one node
B

Hierarchical Tree (Manager → Team Leads → Specialists)

Three-tier hierarchy. Manager routes to domain Team Leads (Coach Lead, Assistant Lead, etc.) → Leads coordinate their specialist teams → Results bubble up.

Pros

  • Scales better — Manager only talks to ~3-5 Leads
  • Domain isolation — Coach team independent from Assistant team
  • Leads can parallelize within their domain

Cons

  • More complex routing logic
  • Extra hop adds latency
  • Harder to coordinate cross-domain tasks
C

Blackboard Architecture (Shared State + Orchestrator)

Shared working memory (the "blackboard") holds request state. Orchestrator manages flow. Specialists read from and write to the blackboard. Orchestrator decides which specialist runs next based on blackboard state.

Pros

  • Maximum flexibility — specialists are decoupled
  • Easy to add conditional workflows
  • State is always inspectable/debuggable
  • Natural fit for iterative refinement

Cons

  • Blackboard schema becomes critical
  • Requires careful state management
  • More complex to implement correctly

My Recommendation

Start with Approach A (Centralized Orchestrator) for MVP, but design the interfaces to support evolving into Approach C (Blackboard).

Why: Your current Triage → Coach/Assistant → DAG → Reviewer flow is already a simpler version of A. The Manager agent replaces Triage, and you can incrementally add blackboard patterns without a full rewrite. This gives you a clear migration path.
